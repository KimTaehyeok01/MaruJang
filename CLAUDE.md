# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

MaruJang is a B2B platform connecting traditional-market merchants (상인) in Jungnang-gu with food-ingredient
wholesalers (도매업체). Suppliers register inventory and pricing; merchants search/filter and place orders.
This is a learning/portfolio project — the user must be able to explain every design decision (interview prep),
and is a complete beginner in React.

The repo is split into `backend/` (Spring Boot, Gradle) and `frontend/` (React, Vite). Entities and repositories
for all 7 domains exist; service/controller layers and most React screens are still placeholders to be built.

## Architecture

- **Backend**: Spring Boot 3.3, Java 21 (Gradle toolchain — local JDK may be older; install JDK 21 to actually
  build/run), JPA, QueryDSL (Jakarta classifier, for dynamic search/filter queries), Spring Security with
  OAuth2 login (Google/Kakao) + JWT issued on successful login.
- **Frontend**: React (plain JavaScript, not TypeScript) + Vite + Tailwind CSS, `react-router-dom` for routing.
- **DB**: PostgreSQL.

Package structure is domain-driven, not layer-driven: `backend/src/main/java/com/marujang/domain/{user,merchant,
supplier,item,inventory,order}/{entity,repository}`, with cross-cutting security/config under
`backend/src/main/java/com/marujang/global/{config,security}`. Mirror this in the frontend: each domain gets its
own folder under `frontend/src/domains/{auth,merchant,supplier,item,inventory,order}/{api,components,pages}`,
with shared axios instance / auth context under `frontend/src/common/`.

Planned flow: ERD (done, see entities below) → build backend API → wire up frontend last. Backend should be
functionally complete (or close to it) before frontend work starts.

### Core entities

- `User` — common account, `role`: `MERCHANT` / `SUPPLIER`, table name `users` (reserved word)
- `Merchant` — 1:1 with `User` via FK (not JPA inheritance); market name, business type, phone
- `Supplier` — 1:1 with `User` via FK; company name, business number, phone
- `Item` — shared ingredient catalog: name, category, unit
- `Inventory` — per-`Supplier`/`Item` quantity + price; unique on `(supplier_id, item_id)`
- `Order` — merchant-initiated **and tied to a single supplier** (`Order.supplier_id`); `status`: REQUESTED /
  CONFIRMED / COMPLETED / CANCELLED. The single-supplier rule is enforced in the service layer, not the DB —
  when adding an `OrderItem`, verify `inventory.supplier_id == order.supplier_id`.
- `OrderItem` — references `Inventory` (not `Item` directly, since price/supplier depend on it), plus
  `price_snapshot` (order-time price copy, so later price changes don't alter past order totals)

`backend/src/main/resources/db.sql` has the full DDL as a human-readable reference; it is **not** auto-executed —
dev schema is generated by `spring.jpa.hibernate.ddl-auto: update`.

## Scope constraints (deliberate — do not add beyond this)

These limits exist because the target skill level is Spring Boot CRUD + JPA basics + Security basics on the
backend, and first-time React on the frontend. Prefer the simpler option even if technically inferior:

- No WebSocket — use refresh/polling for anything that looks like "real-time" updates.
- No Redis distributed locks, no message queues.
- Concurrent inventory updates: plain `@Transactional` only, no custom locking.
- Search/matching: QueryDSL dynamic queries (conditional filtering) — nothing more elaborate.
- No state-management library (Redux, etc.) on the frontend — `useState` / `useContext` only.

## Working with this codebase

- Follow the order: ERD → backend API → frontend, unless the user says otherwise.
- When introducing a new React pattern for the first time (a new component, a new API-call pattern), explain
  *why it's structured that way* (why the component is split there, why state lives where it does) rather than
  just what the code does. Once a pattern has been explained, later repeats can be left to the user to write.
- For inherently complex pieces (QueryDSL dynamic queries, Spring Security config, React state flow), it's fine
  to write the code, but add line-by-line comments explaining what each part does.
- If the user indicates they don't understand something, stop and explain the concept before continuing —
  don't just push forward through the task list.

## Commands

No Gradle wrapper is checked in yet (no local Gradle install to generate `gradle-wrapper.jar` — opening
`backend/` in IntelliJ will offer to set one up, or run `gradle wrapper` once Gradle is available).

Backend (from `backend/`):
- `gradle bootRun` — run the API on port 8080
- `gradle build` — compile + run tests
- `gradle test` — run tests only

Frontend (from `frontend/`):
- `npm install` — install dependencies (not yet run)
- `npm run dev` — Vite dev server on port 5173, proxies `/api` to `localhost:8080`
- `npm run build` — production build

Secrets (JWT signing key, OAuth2 client id/secret) live in `backend/src/main/resources/application-secret.yml`,
which is gitignored and currently has placeholder values — fill in real values before running OAuth2 login.
